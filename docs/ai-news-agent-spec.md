## 1. 概要

### 背景・課題

AI 技術の進化が速く、毎日大量のニュースが世界中で発信されている。X のブックマークを週 1 で斜め読みするワークフローでは情報の偏りやキャッチアップ漏れが発生しており、MCP のような重要技術のキャッチアップが数ヶ月遅れることがある。

現在は Claude Code の Skills で海外テックニュースを取得しているが、**1 回あたり約 10 分かかり実用性が低い**。また翻訳が要約ベースになってしまい、**原文の完全な日本語翻訳が得られない**問題がある。

### このエージェントが解決すること

- 世界中の AI 関連ニュースを自動収集し、重要度順に整理する
- 英語記事を**原文構造を保った完全な日本語翻訳**で提供する（技術用語は英語のまま保持）
- 重複記事を除去して情報密度を高める
- 結果を Markdown ファイルにまとめ、手元で斜め読みできる形にする
- **ポッドキャスト風の音声ダイジェストを生成し、朝の準備中や散歩中に聞ける形にする**
- **毎日・毎週、一定の量と品質のニュースを安定して提供する**
- **現在の Skills ベースのワークフロー（約 10 分）を 2〜3 分に短縮する**

### ゴール（最終形）

Claude Code / Codex CLI の TUI 上で対話的に情報ニーズを明確にし、その結果を元に ai-news-agent が内部でエージェンティックに情報収集・評価・翻訳・整形を実行し、**ダイジェスト md + 音声ダイジェスト**を生成する。

**例**:
```
> 今日はどんな情報が欲しいですか？
→ Hacker News で今盛り上がっている AI 関連のニュース、ここ 1 週間分

> 取得件数は？（デフォルト: 10 件）
→ 5 件でいい、今日は忙しいから

→ [ai-news-agent 起動 → Hacker News から情報収集 → 完全翻訳 → md + 音声生成]
```

```
> 今日はどんな情報が欲しいですか？
→ 仕事に直結するライブラリの更新情報が欲しい

> 最近使っているライブラリを教えてください
→ LangGraph, Cohere, Elasticsearch

> 更新情報のみ、それとも使い方の記事も？
→ 更新情報のみ

→ [ai-news-agent 起動 → 公式 docs・GitHub Releases から情報収集 → 完全翻訳 → md + 音声生成]
```

## 4. フォールバック付き多段フェッチ戦略

固定ソースから取得した情報が「量・質」の基準を満たさない場合、追加ソースへ段階的にフォールバックする。これにより毎日・毎週、安定した量と品質のダイジェストを生成できる。

```
[Step 1] 固定ソース取得（Zenn, Qiita, Hacker News など）
         ↓
         品質・量チェック（LLM-as-judge）
         ┌─ 十分 → [Step 5] へ
         └─ 不足 ↓

[Step 2] 抽象度レイヤーに対応する追加ソース取得（Web 検索など）
         ↓
         品質・量チェック
         ┌─ 十分 → [Step 5] へ
         └─ 不足 ↓

[Step 3] さらに広範なソースを追加（Multi-Query 展開）
         ↓
         品質・量チェック
         ┌─ 十分 → [Step 5] へ
         └─ n 段階まで繰り返し

[Step 5] 最終フィルタリング・整形 → md + 音声生成
```

**品質・量の判断基準（内側エージェントが担当）**:
- 記事数が `limit` に対して著しく不足していないか
- スコアリングで「高・中」評価の記事が一定数あるか
- ソースの多様性（同一サイトに偏っていないか）

## 5. TUI 対話フェーズ設計

実行前に Claude Code / Codex CLI（外側エージェント）がユーザーと対話し、情報ニーズを明確にする。

### 対話の流れ

```
今日はどんな情報が欲しいですか？
  → [自由入力]
  例: "Hacker News で今盛り上がっている AI 関連のニュース、ここ 1 週間分"
  例: "今日のトレンドニュースを 10 本"
  例: "仕事に直結するライブラリの更新情報"

（入力に応じて追加質問を動的に生成）
例:
  ・「仕事直結」と入力した場合 → 使用ライブラリ・フレームワークを確認
  ・「世界の流れ」と入力した場合 → 特定トピックに絞るか確認
  ・「新ツール」と入力した場合 → 言語・領域の絞り込みを確認
  ・媒体指定あり → そのまま採用、追加質問は最小限

取得件数の目安は？（デフォルト: 10 件）
  → [入力またはスキップ]
  ※ ユーザーの気分や忙しさに応じて柔軟に変更可能（3〜20 件程度）

音声ダイジェストも生成しますか？（デフォルト: Yes）
  → [Yes / No]

確認: [要約された情報ニーズ]でよいですか？
  → [Yes / 修正]
```

### 対話結果の渡し方

対話で得た情報を構造化して内側エージェントに渡す：

```typescript
interface NewsRequest {
  query: string;           // 例: "LangGraph Cohere Elasticsearch 更新情報"
  abstractionLayer: "macro" | "middle" | "micro";
  targetSources?: string[];// 例: ["github_releases", "official_docs"]
  date: string;            // 例: "today" | "past_week"
  limit: number;           // 例: 10（3〜20 の範囲で柔軟に指定可能）
  userIntent: string;      // 対話の要約（LLM 用コンテキスト）
  generateAudio: boolean;  // 音声ダイジェスト生成の有無
}
```

## 6. 翻訳仕様

### 基本方針

**要約ではなく完全翻訳**を行う。ユーザーが原文を読むのと同等の情報量を日本語で得られることを目指す。

### ルール

| 項目 | 方針 |
|---|---|
| 技術用語 | 英語のまま保持（例: embedding, fine-tuning, RAG, MCP） |
| 固有名詞 | 英語のまま保持（例: LangGraph, Hacker News） |
| 文章構造 | 原文の段落構造・見出し構造を維持 |
| コードブロック | そのまま保持（翻訳しない） |
| 翻訳対象 | 本文全体（掻い摘んだ要約は行わない） |
| 文体 | 自然な日本語（直訳調にならないよう注意） |

### NG パターン

- ❌ 原文を要約して翻訳する
- ❌ 「この記事では〜について述べています」のようなメタ的要約
- ❌ 技術用語を無理に日本語化する（例: ❌「埋め込み」→ ✅「embedding」）

## 7. 音声ダイジェスト仕様

### 概要

収集した記事群の**要約ポッドキャスト風ダイジェスト**を音声ファイルとして生成する。朝の準備中や散歩中にハンズフリーで情報をキャッチアップすることが目的。

### 仕様

| 項目 | 内容 |
|---|---|
| 形式 | MP3 または WAV |
| 長さ | 3〜10 分（記事数に応じて可変） |
| 言語 | 日本語 |
| スタイル | ポッドキャスト風ダイジェスト（記事ごとの要約読み上げ） |
| 構成 | イントロ → 各記事の概要（重要度順）→ クロージング |

### 音声生成の流れ

```
[1] 翻訳済み記事群から音声用スクリプトを生成（LLM）
    - 各記事を 30 秒〜1 分程度の口語体要約に変換
    - 自然な接続詞で繋ぐ

[2] TTS API で音声ファイルを生成
    - 候補: OpenAI TTS / ElevenLabs / Google Cloud TTS

[3] 出力
    - md ファイルと同じディレクトリに音声ファイルを配置
    - md 内にリンクを記載
```

### md との違い

| | md（完全翻訳） | 音声（ダイジェスト） |
|---|---|---|
| 情報量 | 原文と同等 | 要約（概要レベル） |
| 用途 | じっくり読む | ながら聞き |
| 生成コスト | 翻訳のみ | 翻訳 + スクリプト生成 + TTS |

## 8. パフォーマンス目標

### 現状と目標

| 指標 | 現状（Skills ベース） | 目標 |
|---|---|---|
| 10 記事の取得〜md 生成 | 約 10 分 | **2〜3 分** |
| 翻訳品質 | 要約ベース（情報欠落あり） | 完全翻訳 |

### 高速化の方針

- **記事取得の並列化**: 複数ソースへの fetch を `Promise.all` / `Promise.allSettled` で並列実行
- **翻訳の並列化**: 記事ごとの翻訳を並列実行（API rate limit に注意）
- **不要な再取得の回避**: キャッシュ層の導入（同日中の同一 URL はスキップ）
- **ストリーミング出力**: 翻訳完了した記事から順次 md に書き出し（全記事完了を待たない）

## 9. アーキテクチャ概要

### 全体構造：二重エージェント（Mixture of Agents）

```
┌─────────────────────────────────────────────────────┐
│  外側エージェント：Claude Code / Codex CLI（TUI）     │
│                                                     │
│  ① ユーザーと対話し情報ニーズを明確化                 │
│  ② NewsRequest を構築し fetch_ai_news を呼び出す      │
│  ③ 結果の md を表示・音声ファイルのパスを案内          │
└───────────────────┬─────────────────────────────────┘
                    │ MCP tool call（NewsRequest）
                    ↓
┌─────────────────────────────────────────────────────┐
│  内側エージェント：ai-news-agent（MCP サーバー）       │
│                                                     │
│  ニュース収集に特化したエージェントループを実行         │
│  LLM（プランナー）が以下のツールを自律的に呼び出す      │
│                                                     │
│    ├─ fetchRSS        RSS フィードから記事取得         │
│    ├─ webSearch       Web 検索で最新情報を補完         │
│    ├─ fetchTechBlog   テックブログから記事取得          │
│    ├─ checkDuplicate  embedding による重複判定         │
│    ├─ scoreArticle    LLM-as-judge で重要度評価       │
│    │                  ＋ フォールバック判断            │
│    ├─ translateFull   英語 → 日本語 完全翻訳           │
│    ├─ generateScript  音声用スクリプト生成              │
│    ├─ generateAudio   TTS による音声ファイル生成        │
│    └─ writeMarkdown   md ファイル出力                 │
│                                                     │
│  品質・量が基準を満たすまでフォールバックループを実行   │
│  十分な情報が集まったと判断したら md + 音声を返す       │
└─────────────────────────────────────────────────────┘
```

### 内側エージェントの処理フロー

```
[1] リクエスト受信
    外側から NewsRequest（query / abstractionLayer / limit 等）を受け取る

        ↓

[2] ソース選択
    abstractionLayer に基づき固定ソース群を決定

        ↓

[3] 情報収集（フォールバックループ）【並列実行】
    fetchRSS / webSearch / fetchTechBlog を並列実行
    → scoreArticle で品質・量を評価
    → 不足なら追加ソースで再取得（Multi-Query）
    → n 段階まで繰り返す

        ↓

[4] 正規化
    タイトル・URL・本文・日時・ソース名を統一フォーマットに変換

        ↓

[5] 重複除去
    checkDuplicate でコサイン類似度による重複を検出・統合

        ↓

[6] 重要度スコアリング
    scoreArticle（LLM-as-judge）が各記事を評価
    （技術的重要度・新規性・関心領域との関連度）

        ↓

[7] 完全翻訳【並列実行】
    translateFull で英語記事を日本語に完全翻訳
    （技術用語は英語保持、原文構造維持、要約しない）

        ↓

[8] Markdown 出力
    writeMarkdown で重要度順に並べた md を生成

        ↓

[9] 音声ダイジェスト生成（generateAudio: true の場合）
    generateScript → generateAudio で音声ファイルを生成
    md 内に音声ファイルへのリンクを追記

        ↓

[10] 結果返却
     md テキスト + 音声ファイルパスを外側エージェントに返す
```

### MCP サーバーのインターフェース

```typescript
server.tool("fetch_ai_news", {
  query: z.string(),
  abstractionLayer: z.enum(["macro", "middle", "micro"]),
  targetSources: z.array(z.string()).optional(),
  date: z.string(),
  limit: z.number().min(3).max(20),
  userIntent: z.string(),
  generateAudio: z.boolean().default(true),
}, async (request) => {
  const agent = new NewsAgent(request);
  const result = await agent.run();
  return {
    content: [
      { type: "text", text: result.markdown },
      ...(result.audioPath
        ? [{ type: "text", text: `🎧 音声ダイジェスト: ${result.audioPath}` }]
        : []),
    ],
  };
});
```

## 12. 技術スタック（候補）

| 役割 | 候補 | 備考 |
|---|---|---|
| 言語 | TypeScript | togu と同スタック |
| LLM（内側エージェント） | Claude API（claude-sonnet-4-6） | Tool use サポート |
| MCP サーバー実装 | `@modelcontextprotocol/sdk` | 公式 TypeScript SDK |
| Web 検索 | Tavily API / Perplexity API | Phase 2 で検討 |
| RSS パース | `rss-parser` npm | Phase 1 から使用 |
| Embedding（重複除去） | Voyage AI / OpenAI embedding | Phase 2 で導入 |
| TTS（音声生成） | OpenAI TTS / ElevenLabs / Google Cloud TTS | Phase 4 で選定 |
| 出力 | fs（Node 標準） | md + 音声ファイル書き出し |

## 13. 成功基準

- **Phase 1**: 1 コマンドでその日のダイジェスト md が生成できる。**完全翻訳の品質が満足レベル**。**10 記事で 3 分以内**。
- **Phase 2**: 重複記事が目視で明らかに減り、重要度順が体感で正しい。量・質が不足した日もフォールバックで補完されている。テックブログからの収集が機能している。
- **Phase 3**: TUI 上で対話 → 情報ニーズ明確化 → ニュース収集 → md 生成が完結する。媒体・本数・トピックを柔軟に指示できる。
- **Phase 4**: 音声ダイジェストが生成され、朝の準備中や散歩中に聞ける品質になっている。
- **Phase 5**: 「X ブックマークを週 1 で見る」習慣が不要になる。

## 14. ポートフォリオとしての価値

- 二重エージェント構造の設計と実装（MoA: Mixture of Agents）
- MCP サーバーの自作経験（2026 年時点で差別化になる）
- LLM-as-judge による評価指標設計（仕事の RAG 評価と同じ構造）
- embedding を使った重複除去（RAG のチャンク品質管理と同じ技術）
- **完全翻訳パイプラインの設計（要約ではなく原文忠実翻訳）**
- **音声ダイジェスト生成（TTS 統合）**
- **並列処理によるパフォーマンス最適化設計**
- **フォールバック戦略を持つ品質保証ループの設計**
- **TUI 対話 → 動的ソース選択 → エージェント実行の E2E 設計**
- TypeScript での CLI → MCP サーバー移行の設計判断

Helpfeel・カオナビの面接軸「評価インフラの設計経験」および「エージェント設計経験」に直結する実装になる。
