# ai-news-agentプロジェクトで設計力とコードレビュー力を鍛える方法

**作成日**: 2026-03-22
**対象**: ミドルレベルで「設計・レビューに自信がない」人

## 設計力の正体と鍛え方
設計力 = 「AIに正しい指示を出す判断力」
（完璧に1人で書く力ではない）

### 鍛える3ステップ（ai-news-agentで実践）
1. **写経**（Week 1-2で実施）
   - LangGraph.js / MCPの公式exampleを全部動かす
   - 「なぜこのState定義にしたか？」「このedgeはいつ発火するか？」をメモ

2. **手書き設計**（毎週末必須）
   - Draw.io or 紙に「State → Nodes → Edges」を描く
   - 仕様書の処理フロー（フォールバックループなど）を自分でグラフ化

3. **AIに指示を出す練習**
   - 「このState定義でfetchRSSツールを追加して」とClaude Codeに指示
   - 指示が出せないときは「手書きグラフを見ながらプロンプトを書く」

**設計力向上のチェック項目**（毎回自分に聞く）
- Stateは本当に最小か？
- フォールバックはグラフのどのedgeで処理するか？
- MCPツールのschemaはLLMが安全に呼べるか？

## コードレビュー力の鍛え方
レビュー力 = 「仕様書に書いた意図が守られているかをチェックする力」

### ai-news-agent専用レビュー checklist（これをコピーして使用）
毎回AI生成コードに対して以下を適用してください（最初は上位5項目だけでもOK）：

1. **State管理**：LangGraphのStateが最小で型安全か？
2. **MCP準拠**：zod schemaが厳密で、LLMが理解しやすいか？
3. **耐障害性**：フォールバックにmax_retryとPromise.allSettledがあるか？
4. **パフォーマンス**：並列化が効いていて3分目標を守れそうか？
5. **翻訳ルール**：技術用語は英語保持、要約していないか？
6. **単一責任**：1関数1役割か？
7. **Observability**：Langfuseログ挿入可能か？
8. **テスト性**：単体テストが書きやすい構造か？
9. **セキュリティ**：APIキー漏洩の危険性はないか？
10. **コメント**：「なぜこの設計か」が書いてあるか？

### 実践方法（一人PR活用）
- featureブランチで実装 → PR作成（Draft）
- PR画面で上記checklistを全部適用し、自分でコメントを残す
- 「Approve → Merge」する（これが自己レビュー習慣）
- 毎回「このPRで設計判断したポイント」をPR本文に書く

## 習慣化のコツ
- 学習ノート（Notion）に「今日の設計判断」と「レビューで直した箇所」を必ず1行書く
- Phase終了ごとに「このプロジェクトで設計力がどう上がったか」を自分に振り返る
- 完成後、GitHub READMEに「開発プロセス（設計・レビュー方針）」セクションを追加（転職で最強アピール）

この2つの力を鍛えれば、**ai-news-agent完成時には「Seniorエンジニアの設計・レビューができる人」**として堂々と言えます！

進捗や詰まったポイントはいつでもIssueやここに書いてください。一緒にSeniorレベルまで引き上げましょう！
